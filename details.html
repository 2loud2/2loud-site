The Details

Everything under the hood — the technology, the architecture decisions, the roadmap, and the hard questions. This page is for the people who want to look closely.

The Technology

2Loud uses Google's YAMNet audio classification model, running locally in your browser via TensorFlow.js. YAMNet was trained on AudioSet — over 2 million human-labeled 10-second sound clips covering 521 audio event classes.

The extension captures audio from your active tab using Chrome's Web Audio API. That audio is processed in real time through YAMNet, which outputs confidence scores for each of its 521 classes. When a trigger sound exceeds the user's set threshold, the extension ducks the volume using a GainNode — no audio is ever recorded, stored, or transmitted.

Everything happens on your machine. The model weights are downloaded once and cached. There is no server call during operation. The only network request the extension ever makes is an optional, anonymous feedback event if you choose to report a miss or false alarm.

How the Filter Actually Works

Audio comes through your browser tab. The extension sits between the source and your speakers, processing in real time:

1. Capture. Chrome's tabCapture API routes the tab's audio stream to the extension.

2. Buffer. A short pre-buffer (adjustable) delays the audio slightly, giving the model time to classify before the sound reaches you. Think of it like broadcast TV delay.

3. Classify. YAMNet processes audio frames and returns confidence scores per class. The extension maps specific classes to user-defined trigger categories (doorbell, gunshot, screaming, barking).

4. Duck. When a trigger is detected above the user's sensitivity threshold, the GainNode smoothly reduces volume to the level set for that trigger. When the sound passes, volume fades back up.

Each trigger has its own independent volume slider — screams at 50%, gunshots at 0%, doorbells at 0%. It's not a binary mute. It's a sound mixer for sensitivity.

Roadmap

Phase 1 — Proof of Concept

Chrome extension with local YAMNet classification, four trigger categories, per-trigger volume sliders, basic popup UI. Testing with friends. This is where we are now.

Phase 2 — Feedback Loop

Add thumbs up/down feedback mechanism. Anonymous event logging (sound category, confidence score, user verdict, content timestamp). No accounts, no tracking. Begin building the dataset.

Phase 3 — Content Trigger Map

Searchable index of media content by trigger density. Type in a movie, see the trigger timeline. Parents check before pressing play. Dog owners pick films by doorbell count.

Phase 4 — Research Infrastructure

Session codes for researchers. Live dashboard. Public API for academic and commercial queries. Open dataset release.

Phase 5 — Scale

Firefox extension. Desktop app with system-wide audio processing. Partnerships with streaming platforms. Impact fund conversations. The extension becomes infrastructure.

Session Codes: Research Without Surveillance

A researcher, therapist, or study coordinator generates a session code — a short, temporary identifier tied to a study, not a person. Participants enter the code in the extension before watching. Their anonymous feedback events get tagged to that session instead of flowing into the general pool.

A misophonia researcher creates a session, shares the code with 200 participants. Each participant watches the assigned content. The researcher sees the aggregate: which sounds were flagged, at what confidence, how many misses and false alarms.

No participant creates an account. No emails. No names. The session code links data to a study, not to a person. One participant could enter the code on three different devices and you'd never know it was the same person.

This turns the extension into research infrastructure. Not as a side effect — as a deliberate, designed capability.

The Chain of Insights

Layer 1: The extension works. Trigger sounds are detected and filtered. Users get a quieter experience. That's the consumer product.

Layer 2: Accuracy data. Thumbs up/down feedback reveals where the model is right and wrong across different platforms, content types, and audio encodings. This makes the classifier better.

Layer 3: Sensitivity calibration. The gap between model confidence and human reaction maps to a sensitivity spectrum. An 87% confidence gunshot that humans consistently flag as "not bothering me" tells a different story than a 60% confidence doorbell that everyone flags as a miss.

Layer 4: Novel sound discovery. When users hit "missed" at timestamps where the model detected nothing, they're revealing sounds — or experiences — that no dataset has captured. Sounds below classification thresholds. Cross-modal triggers. Frequencies. Textures.

Layer 5: The unknown. With enough data, patterns will emerge that nobody predicted. That's the point of open research infrastructure — you don't have to know what you'll find. You have to build the system that can find it.

Known Risks and Honest Limitations

The first fraction of a sound gets through. The AI needs to hear a sound before it can classify it. The pre-buffer helps, but it adds latency. For dogs that react to the very onset, this might not be fast enough. Your feedback determines whether we prioritize this.

False positives will happen. The model will sometimes duck volume during normal dialogue because it thought someone was screaming. Every false alarm report tells us where to tune.

Platform variability. Audio encoding differs across Netflix, YouTube, Disney+, and others. Model accuracy may vary by platform. We won't know the full picture until users are testing across all of them.

Streaming DRM restrictions. Some platforms may restrict the audio capture APIs the extension relies on. This is a technical risk that could limit which services work. We're transparent about it.

Sustainability. A non-profit with no ad revenue needs alternative funding. Grants, API licensing, and donations are the plan. If those don't materialize, the project faces the same pressure as any unfunded effort.

True Transparency: "We Can't," Not "We Won't"

Most companies publish a privacy policy that says "we won't sell your data." Those are promises. Promises can be broken.

We're publishing the architecture instead.

What you'd need to identify a user:

  • IP address
  • Device fingerprint or unique identifier
  • Account identity (email, login, name)
  • Session ID linking multiple events
  • Viewing history tied to a session
  • Time of day (user's local time)
  • Location or timezone
  • Demographic information
  • Behavioral profile stored server-side

What we actually collect:

  • Sound category (e.g., scream, doorbell)
  • Model confidence score (e.g., 87%)
  • User feedback (miss, false alarm, or none)
  • Timestamp within the content
  • Content identifier (title or URL)
  • Platform (YouTube, Netflix, etc.)
The gap between these two sets is the entire point. There are approximately seven layers of data between what we collect and what you'd need to identify a person — and we don't collect any of them.

This isn't a policy decision that a future board could reverse. It's an architecture decision. The system doesn't have a user table. There's no field to put your name in even if someone wanted to.

Legal Precedent

The Family Movie Act of 2005 established that technology which allows viewers to skip or mute content during home viewing does not violate copyright, as long as it doesn't create an altered copy of the work.

ClearPlay, which filters objectionable content from movies, operated under this framework. 2Loud applies the same principle: we're modifying the viewer's real-time experience, not the content itself. No altered copies are created, stored, or distributed.
