The Details

Everything under the hood — the technology, the architecture decisions, the roadmap, and the hard questions. This page is for the people who want to look closely.

The Technology

2Loud uses Google's YAMNet audio classification model, running locally in your browser via TensorFlow.js. YAMNet was trained on AudioSet — over 2 million human-labeled 10-second sound clips covering 521 audio event classes.

The extension captures audio from your active tab using Chrome's Web Audio API. That audio is processed in real time through YAMNet, which outputs confidence scores for each of its 521 classes. When a trigger sound exceeds the user's set threshold, the extension ducks the volume using a GainNode — no audio is ever recorded, stored, or transmitted.

Everything happens on your machine. The model weights are downloaded once and cached. There is no server call during operation.

How the Filter Actually Works

1. Capture. Chrome's tabCapture API routes the tab's audio stream to the extension.

2. Buffer. A short pre-buffer delays the audio slightly, giving the model time to classify before the sound reaches you. Think broadcast TV delay.

3. Classify. YAMNet processes audio frames and returns confidence scores per class, mapped to user-defined trigger categories.

4. Duck. When a trigger is detected above threshold, the GainNode smoothly reduces volume to the level set for that trigger. When the sound passes, volume fades back up.

Each trigger has its own independent volume slider. It's not a binary mute — it's a sound mixer for sensitivity.

Roadmap

Phase 1 — Proof of Concept

Chrome extension with local YAMNet classification, four trigger categories, per-trigger volume sliders. Testing with friends. This is where we are now.

Phase 2 — Feedback Loop

Thumbs up/down feedback. Anonymous event logging. No accounts, no tracking. Begin building the dataset.

Phase 3 — Content Trigger Map

Searchable index of media content by trigger density. Type in a movie, see the trigger timeline.

Phase 4 — Research Infrastructure

Session codes for researchers. Live dashboard. Public API. Open dataset release.

Phase 5 — Scale

Firefox extension. Desktop app. Streaming platform partnerships. The extension becomes infrastructure.

Session Codes: Research Without Surveillance

A researcher generates a session code — a short, temporary identifier tied to a study, not a person. Participants enter the code in the extension before watching. Their anonymous feedback gets tagged to that session instead of flowing into the general pool.

No participant creates an account. No emails. No names. The session code links data to a study, not to a person. One participant could enter the code on three different devices and you'd never know it was the same person.

This turns the extension into research infrastructure — not as a side effect, but as a deliberate, designed capability.

The Chain of Insights

Layer 1: Trigger sounds are detected and filtered. That's the consumer product.

Layer 2: Feedback reveals where the model is right and wrong across platforms and content types. The classifier gets better.

Layer 3: The gap between model confidence and human reaction maps to a sensitivity spectrum nobody has measured before.

Layer 4: Users flag moments where the model detected nothing — revealing sounds or experiences no dataset has captured.

Layer 5: With enough data, patterns emerge that nobody predicted. That's the point of open research infrastructure.

Known Risks and Honest Limitations

The first fraction of a sound gets through. The pre-buffer helps, but for dogs that react to the very onset, it might not be fast enough.

False positives will happen. The model will sometimes duck volume during normal dialogue. Every false alarm report tells us where to tune.

Platform variability. Audio encoding differs across Netflix, YouTube, Disney+. Accuracy may vary. We won't know the full picture until users are testing across all of them.

Streaming DRM restrictions. Some platforms may restrict the audio capture APIs we rely on. This is a real technical risk.

Sustainability. A non-profit with no ad revenue needs alternative funding. Grants, API licensing, and donations are the plan.

"We Can't," Not "We Won't"

Most companies say "we won't sell your data." Those are promises. Promises can be broken. We're publishing the architecture instead.

What you'd need to identify a user:

  • IP address
  • Device fingerprint or unique identifier
  • Account identity (email, login, name)
  • Session ID linking multiple events
  • Viewing history tied to a session
  • Time of day (user's local time)
  • Location or timezone
  • Demographic information
  • Behavioral profile stored server-side

What we actually collect:

  • Sound category (e.g., scream, doorbell)
  • Model confidence score (e.g., 87%)
  • User feedback (miss, false alarm, or none)
  • Timestamp within the content
  • Content identifier (title or URL)
  • Platform (YouTube, Netflix, etc.)

The gap between these two sets is the entire point. There are approximately seven layers of data between what we collect and what you'd need to identify a person — and we don't collect any of them. This isn't a policy decision. It's an architecture decision. The system doesn't have a user table.

Legal Precedent

The Family Movie Act of 2005 established that technology which allows viewers to skip or mute content during home viewing does not violate copyright, as long as it doesn't create an altered copy. ClearPlay operated under this framework. 2Loud applies the same principle.
